Fair N Square

Multi-repo status of the open-source expense-splitter — a Gen-1 → Gen-2 pivot, what's built (auth + contracts), and the core product still to build.

~13 repos · fair-n-square-co SvelteKit · Go · protobuf · PostgreSQL Roadmap: 351 pts / 10 epics Target: Q4 2025
Headline decision

There was a generational pivot, and the tracking artifacts point at the old generation. A 2024 stack (Flutter app + Go transactions on GORM + Firebase auth) was largely replaced in 2025 by a new stack (SvelteKit webapp + Better Auth, Go ledger on sqlc, shared apis protobuf). The GitHub Project board (#3) still only tracks the 2024 issues — treat docs/ROADMAP.md, not the board, as the source of truth.

03Data models that exist today

Only two schemas are actually implemented. Everything else (groups, expenses, balances, settlements) is still only a contract or a roadmap line. The webapp's application schema is literally export const schema = {} — only the Better Auth tables exist there.

webapp · src/lib/server/db/betterauth/schema.ts-- user.id is referenced by session/account/passkey (all ON DELETE CASCADE)
CREATE TABLE "user" (
  id             text PRIMARY KEY,
  email          text NOT NULL UNIQUE,
  name           text NOT NULL,
  email_verified boolean,
  image          text
);
CREATE TABLE session (
  id, user_id REFERENCES "user"(id) ON DELETE CASCADE, token UNIQUE, expires_at
);
CREATE TABLE account (
  id, user_id REFERENCES "user"(id), provider_id,
  password,      -- set for email/password
  access_token   -- set for OAuth
);
CREATE TABLE passkey ( id, user_id, public_key, credential_id );
CREATE TABLE jwks    ( id, public_key, private_key );  -- RS256 keys, sign API JWTs
ledger · db/migrations/20251027225213_friends.sqlCREATE TYPE friend_status AS ENUM ('pending', 'accepted', 'blocked');
CREATE TABLE persona (
  id         uuid PRIMARY KEY,   -- external system reference (auth user)
  created_at timestamptz
);
CREATE TABLE friend (
  id             uuid PRIMARY KEY,
  persona_id_1   uuid REFERENCES persona(id),
  persona_id_2   uuid REFERENCES persona(id),
  status         friend_status,  -- pending | accepted | blocked
  name_persona_1 text,
  name_persona_2 text            -- UNIQUE on normalized (p1, p2) pair
);

04API contracts (apis repo)

The protobuf contracts are ahead of the implementations — defined and code-generated (Go + TS), but most backend handlers are still stubs. Paths are the connectRPC routes.

POST/fairnsquare.service.user.v1alpha1.UserService/CreateUser

UserService also exposes GetUser and IsUsernameAvailable.

POST/fairnsquare.service.user.v1alpha1.GroupService/CreateGroup

GroupService: CreateGroup, ListGroups, UpdateUsersInGroup. No backend implements these yet in Gen 2.

POST/fairnsquare.service.ledger.v1alpha1.FriendService/ManageFriend

FriendService: ManageFriend, ListFriends. Registered in ledger but handlers return empty responses. Nearest-term implementable RPC, since the schema exists.

POST/fairnsquare.service.transaction.v1alpha1.TransactionService/CreateTransaction

TransactionService: CreateTransaction, ListTransactions. Carries description, google.type.Money amount, type (PAYMENT|SETTLEMENT), per-user paid/owed shares. Contract only.

Bottom line

Auth and the architectural skeleton are genuinely done and solid. The actual expense-splitting product — groups, expenses, balances, settlement — has essentially not been built yet in the current generation; only the contracts and the ledger's friend schema exist. Highest-leverage next move: implement one vertical slice end-to-end (proto → ledger handler → webapp schema/route/UI), starting with Friends.
