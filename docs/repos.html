5 · Multi-repo vs monorepo

Your ADR-2 already accepts "a lot of boilerplate to manage multiple repos" and "longer to set up CI/CD for multiple services" as negatives taken on for the learning value. That's the deciding factor.

  • Keep multi-repo if learning multi-service CI/CD, cross-repo proto publishing, and independent deploys is the goal — higher learning, higher overhead.
  • Go monorepo to ship the MVP faster — eliminates cross-repo proto-versioning pain (brutal for 1–2 people) while still building independent service containers.
Concrete recommendation: monorepo for code (core, auth, webapp, apis, infra, e2e as top-level dirs) + keep docs separate. ~90% of the architectural learning without the cross-repo coordination tax that dominates solo work. If you specifically want the cross-repo contract-publishing lesson, keep apis split out and monorepo the rest.
