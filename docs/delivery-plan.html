Fair N Square — MVP & Stages

The bigger picture, a crisply-scoped MVP, and the dependency-ordered stages that get us there and beyond — distilled from the ROADMAP, ADRs, and the live Jira board.

1 · The Bigger Picture

What we're building. Fair N Square is an open-source, free expense-splitting app (a Splitwise alternative) built deliberately with enterprise-grade architecture. There are two intertwined goals worth keeping distinct:

  • Product goal — ship a genuinely useful expense splitter: groups, friends, expenses, splits, balances, settle-up, debt simplification.
  • Learning goal — build complex systems with real boundaries: a React BFF, a separate Auth service and Core service, gRPC/connectRPC, Postgres, CI/CD, observability — deployed for real on free tiers.

These goals pull against each other on speed. The strategy below resolves that: honor the real architecture, but cut a thin vertical slice through it first, then harden and widen.

2 · MVP Definition

MVP thesis: a user can sign up, connect with friends, create a group, record shared expenses (split equally / by exact amounts / by percentage), see who owes whom, simplify the debts, and settle up — running on the real two-service gRPC architecture, deployed publicly.

In scope for MVP

AreaIncluded
IdentityGoogle login via WorkOS AuthKit, sessions in the BFF, basic profile
AuthorizationSimple ownership/membership checks (am I a member of this group?). Not full ReBAC.
Social graphFriend requests (pending/accepted/rejected), groups, add/remove members
ExpensesAdd/edit/delete expense; split equal / exact / percentage; notes
MoneyBalance engine (who owes whom), settle-up, debt simplification
Multi-currencyExpenses in different currencies with conversion stage 5
Attachments & commentsReceipt photo/PDF upload + comments on expenses stage 5
Group todosGroup todo lists with task assignment stage 5
PlatformContainerized, deployed to AWS (ECS Fargate), basic logging + health checks, core E2E tests
The last four feature rows were explicitly requested as part of the MVP. They are real work, so they're sequenced into a dedicated MVP-Features stage after the core money loop is proven — they extend the core rather than block it.

Explicitly OUT of MVP → deferred to Post-MVP

  • Full fine-grained authorization / ReBAC (OpenFGA / Permit.io) — MVP uses simple membership checks
  • Distributed tracing, full metrics dashboards, alerting/on-call, SLAs
  • Advanced reporting, analytics, CSV/PDF export · transaction categories
  • Push notifications, in-app chat · native mobile apps, offline support, MCP server
  • Performance/load testing, formal security audit & penetration testing

MVP success criteria

  • Multiple real users can sign up and use the system end-to-end.
  • The full money loop (expense → balance → simplify → settle) is correct and verifiable.
  • The app is publicly reachable, mobile-responsive, and survives a redeploy.
  • Core flows are covered by automated E2E tests in CI.

3 · Delivery Stages

Stages are ordered by dependency; each ends at a demoable checkpoint. Stages 0–6 are the MVP.

Stage 0

Foundation completion

unblocks everything
  • Repo hygiene (see repos): create core (fold in ledger), create infra, rename apis → proto, archive/delete dead repos. auth-api is still a "Go API Template" — Stage 1 is greenfield there.
  • Database setup: Auth DB + Core DB schemas, Drizzle/goose migrations, pooling FNS-5
  • Core service scaffold: Go + gRPC/connectRPC server, proto layout, DB wiring FNS-25
  • CI/CD: build + test + lint pipelines per service FNS-6
  • Local stack: docker-compose for Postgres + services FNS-601 (partial)
Checkpoint: every service builds in CI; docker-compose up brings the stack up locally.
Stage 1

Auth & Identity

MVP
  • WorkOS AuthKit in the React BFF: Google login, sessions FNS-9, FNS-11
  • User profile CRUD in Auth service + profile UI FNS-10, FNS-17
  • M2M token service; JWK hosting/validation FNS-13
  • Core ↔ Auth integration layer: token validation, user context FNS-26
Checkpoint: a user signs up, logs in, edits their profile; Core can validate a request's identity.
Stage 2

Groups & Friends

MVP
  • Friends backend (request/accept/reject) + UI FNS-28, FNS-20
  • Groups backend (CRUD, membership, settings, soft-delete) + UI FNS-27, FNS-19
Checkpoint: a user adds a friend and creates a group with members.
Stage 3

Expenses & Splitting

MVP core
  • Expense backend: CRUD, equal / exact / percentage split logic, validation, history FNS-29
  • Expense UI: add/edit/delete, split selector, detail view, list/filters FNS-21
  • Reuse the split algorithm reference (PDF); verify with unit tests. single currency
Checkpoint: a member adds an expense to a group and the split is computed and stored correctly.
Stage 4

Balances & Settlement

MVP core — closes the loop
  • Ledger + balance engine: who-owes-whom, group & per-user balances FNS-30
  • Settlement + debt-simplification algorithm; settle-up flow FNS-31
  • Balance & settlement UI: balances, simplify viz, settle-up, history FNS-22
Checkpoint: the entire money loop works end-to-end on the real architecture.

🏁 Milestone — MVP Core / Alpha

Stage 5

MVP Feature Completion

MVP
  • Multi-currency: per-expense currency + conversion in the ledger extends FNS-29, FNS-30
  • Attachments & comments: file storage selection, receipt upload, comments extends FNS-29, FNS-21
  • Group todos: todo lists + task assignment backend + UI FNS-32, FNS-23
Priority within stage: multi-currency → attachments/comments → todos. Todos and attachments can slip past launch without breaking the core promise.
Stage 6

Production Readiness

MVP launch
Deployment target: AWS (the learning track), provisioned with OpenTofu — see AWS architecture. The MVP runs the minimal slice (ECS Fargate + public ALB + Cloud Map + one RDS Postgres); async/distributed pieces land in P1. Fly.io remains a simpler fallback.
  • Containerize all services; multi-stage Docker builds FNS-601
  • Production infra on AWS via OpenTofu: ECS Fargate, ALB + ACM + Route 53, RDS Postgres (2 DBs), VPC + endpoints, ECR FNS-602 (re-scoped)
  • Env/secrets management (SSM Parameter Store / Secrets Manager) FNS-603
  • Deployment automation + rollback: GitHub Actions + OIDC, tofu plan/apply, ECS circuit-breaker FNS-604
  • Baseline security: rate limiting, validation, CORS, TLS everywhere FNS-703, FNS-702 (partial)
  • Minimal observability: structured logging + health checks/uptime FNS-501, FNS-502 (partial)
  • Core tests: unit (split/ledger) + E2E for critical journeys in CI FNS-57, FNS-59
Checkpoint: publicly deployed, monitored at a basic level, core flows tested.

🏁 Milestone — MVP Launch

4 · Post-MVP Stages

P1 — Hardening & Enterprise Architecture post-mvp

Full distributed-system track in AWS architecture §5 and §8.

  • Async / event-driven (the real distributed leap): EventBridge + SQS + a Worker for balance recalculation & debt simplification, with DLQ + idempotency extends FNS-30/31
  • S3 pre-signed receipt uploads + SNS notification fan-out extends FNS-29
  • Full fine-grained authorization / ReBAC (OpenFGA or Permit.io) FNS-12
  • Distributed tracing (ADOT → X-Ray) + full metrics dashboards + alerting FNS-37, FNS-36, FNS-38
  • Full test pyramid: integration, broader E2E, UAT FNS-58, FNS-60, FNS-61
  • Security audit, SAST/DAST, compliance docs FNS-46, FNS-49, FNS-50
  • Performance/load testing baseline FNS-39, FNS-904

P2 — Advanced Product Features post-mvp

  • Advanced reporting + analytics + CSV/PDF export FNS-52
  • Transaction categories FNS-53
  • Push notifications (Firebase/OneSignal) FNS-54

P3 — Reach post-mvp

  • Native mobile apps (Flutter/React Native), offline support FNS-804
  • MCP server for AI agents

5 · Critical Path & Sequencing

Stage 0 ──► Stage 1 ──► Stage 2 ──► Stage 3 ──► Stage 4 ──► [Alpha]
                                                   │
                                                   ├─► Stage 5 (features)
                                                   └─► Stage 6 (launch) ──► [MVP Launch]
                                                                               │
                                                                  P1 ─► P2 ─► P3
  • Stage 0 is the current bottleneck — Core service and DB schemas don't exist; almost everything depends on them.
  • Authorization is deliberately deferred. Simple membership checks in MVP, full ReBAC in P1 — the biggest scope-saver without hurting the product.
  • Observability is split: "can I see logs and is it up?" is MVP; tracing/alerting/SLAs are P1.
  • Stages 5 and 6 can run partly in parallel once Alpha is reached.

6 · Risks & Mitigations

RiskMitigation
Architecture complexity stalls the productThin vertical slice first (Stages 1–4); defer FGA, tracing, multi-repo polish
Multi-currency complicates the ledger earlyBuild the ledger single-currency in Stage 4; add currency in Stage 5
Solo/learning project, unbounded scopeHard line at "MVP Launch"; everything else is explicitly P1–P3
Free-tier limits on hosting/o11yKeep MVP footprint small (2 services + 2 DBs); pick OSS-friendly free tiers
