# Core Service — Overview

> **Current state as of 2026-06-22.** Edit in place when things change; record *why* in a new ADR.
> See [ADR-2](../../design/records/2-system-design-v1.md) for the system design decisions.

## What it is

The Core Service is a Go modular monolith that owns all business logic: groups, friends, expenses,
settlement, ledger, and todos. It exposes gRPC APIs (via connectRPC) and talks to the Auth Service
for token validation and permissions.

Repository: `github.com/fair-n-square-co/core`

## Module layout convention

Each domain module (`ledger`, `groups`, `friends`, `expenses`, `settlement`, `todos`) follows:

```
internal/<module>/
  api/          ← gRPC handlers + request validation; implements the proto service interface
  service/      ← domain logic; defines the module's public interface (the composability seam)
  repository/   ← DB queries; receives *pgxpool.Pool from internal/core/db
```

**Dependency direction:** `api` → `service` → `repository`. No cross-module package imports —
modules talk to each other through the service interface, not by reaching into each other's packages.

## Proto contracts

Proto definitions live in the `apis` repo (`github.com/fair-n-square-co/apis`). Each module's
`api/` package imports the generated Go stubs and satisfies the gRPC server interface.

## Configuration

Config loading lives next to the entrypoint in `cmd/core/config/` (not under `internal/core/`).
It resolves in **two viper passes into one struct**: the embedded YAML (`config.yml`, or
`config.prod.yml` when `CORE_ENV=production`) loads first, then environment variables override it.
Env vars use the `CORE_` prefix with `_` as the nested-key delimiter, and bind to the struct via
viper's `ExperimentalBindStruct` (so composed sub-configs like `db.DBConfig` are populated through
the same mechanism — e.g. `CORE_DB_CONNSTRING` → `Config.Db.ConnString`). The YAML is
`//go:embed`-ed into the binary, so no config file ships in the Docker image.

Each module/sub-config owns its own config struct (e.g. `db.DBConfig`, `logger.LogConfig`) and is
composed into the top-level `Config`; `server.go` passes the populated sub-config down (e.g.
`db.NewPool(ctx, cfg.Db)`).

> The app reads `CORE_DB_CONNSTRING`. `just migrate-db` is a separate goose CLI that still
> uses `DATABASE_URL`.

## DB layer

DB is a service-level concern, not per-module:

- Pool created at startup via `db.NewPool` (`internal/core/db/`), which pings to fail fast on a bad
  DSN; `*pgxpool.Pool` satisfies the generated `sqlc.DBTX` and is injected into each `repository/`
- Migrations in `db/core/migrations/` (goose)
- Queries in `db/core/queries/` (sqlc); generated code in `internal/core/db/sqlc/`

## Build & run

Requires: `go 1.26`, `just` (`brew install just`), `sqlc`, `goose`; Docker for the compose stack.

```
just build         # go build ./...
just test          # go test ./...
just lint          # golangci-lint run
just run           # build + run cmd/core
just docker-up     # docker compose up --build (app + postgres)
just docker-down   # docker compose down
```
